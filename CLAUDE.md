## Fork context

This repo (`evergreen-connect/php-webassm`) is a maintained fork of upstream `wasmerio/wasmer-php`, modernized for Wasmer 7.x and PHP 8.3+. Published independently, not contributing upstream. Favor changes that serve this project's use case — bumping version support, dropping unused platforms/features, restructuring build tooling.

## What this is

PHP bindings for the [Wasmer](https://github.com/wasmerio/wasmer) WebAssembly runtime. Two layers:

- **`ext/`** — the native PHP extension (C), implementing a procedural API (`wasm_engine_new()`, `wasm_store_new()`, ...) that mirrors the Wasm C API. This is the low-level layer built with `phpize`/`configure`/`make`.
- **`src/`** — a PHP object-oriented wrapper (namespace `Wasm\`) around the procedural API. Every class (`Engine`, `Store`, `Module`, `Instance`, `Func`, `Val`, ...) holds an opaque PHP `resource` in a private `$inner` property, validates it in the constructor via `is_resource()` + `get_resource_type()` (throwing `Exception\InvalidArgumentException` otherwise), frees it in `__destruct()` via the matching `wasm_*_delete()` call, and exposes it via an `inner()` method for other wrapper classes to pass down to procedural calls. New instances are created through static `::new()` factories, not `new Foo()` directly. Follow this exact pattern when adding or modifying a wrapper class.

The extension must be compiled and loaded (`-dextension=ext/modules/wasm.so`) for *any* PHP code (unit tests, examples, doc examples) to run — there is no pure-PHP mock/stub of the native layer.

## Build & test commands

All top-level commands run via `make` (see `Makefile`, which includes `ext/Makefile.frag` after `ext/Makefile` is generated).

```bash
make ext/configure     # phpize + configure the extension (first time / after config.m4 changes)
                        # use PHP_HOME=/path/to/php to target a specific PHP install
make ext/all           # build the extension (ext/modules/wasm.so)
make ext/test          # run the extension's own phpt tests (ext/tests/**)
make ext/examples      # run the procedural-API examples (ext/examples/**)
make ext/clean         # clean extension build artifacts (preserves ext/lib/libwasmer.so)

make test-unit         # run OO interface unit tests (tests/unit, via PHPUnit)
make test-examples     # run OO interface examples (tests/examples, via PHPUnit)
make test-doc-examples # run each examples/*.php file directly; EXAMPLE=<name> to run just one
make test-all          # ext/test + ext/examples + test-unit + test-examples + test-doc-examples

make lint              # php-cs-fixer --dry-run over examples/, ext/examples/, ext/src/, src/, tests/
make documentation     # build phpDocumentor output (.phpdoc/build)
make info              # print `php --ri wasm` (loaded extension info)
make version           # print PHP version in use
```

Run a single PHPUnit test: build the extension first, then invoke PHPUnit directly with the extension loaded, e.g.:
```bash
php -dextension=ext/modules/wasm.so vendor/bin/phpunit --filter testName tests/unit/SomeTest.php
```

`PHP_EXECUTABLE` env var overrides which `php` binary is used everywhere (defaults to `php`); `.atoum.php` respects it too.

CI (`.github/workflows/main.yaml`) runs on Linux and macOS against PHP 8.0/8.1 in this order: `ext/configure` → `ext/all` → `ext/test` → `ext/examples` → `test-unit` → `test-examples` → `test-doc-examples`.

## Architecture notes

- `ext/src/wasm.c` is the extension entry point; `ext/src/api/` holds the procedural implementations split into `api/*.c` (engine, store, config, module, wat), `api/objects/*.c` (func, global, instance, memory, table, trap, val, etc.), and `api/types/*.c` (functype, valtype, externtype, etc.) — each mirroring one concept from the Wasm C API.
- Arginfo headers (`ext/src/wasmer_*_arginfo.h`) are generated build artifacts, listed as prerequisites in `ext/Makefile.frag`'s `all` target — don't hand-edit them if regenerated by the build.
- `src/` has no build step; it's autoloaded via PSR-4 (`Wasm\` → `src/`, `Wasm\Tests\` → `tests/unit`, `Wasm\Examples\` → `tests/examples`) per `composer.json`.
- `Wasm\Vec\*` value types (e.g. `Wasm\Vec\ValType`, `Wasm\Vec\Extern`, `Wasm\Vec\Val`) wrap arrays passed across the procedural boundary — see the README examples for calling conventions between the procedural and OO APIs.
- `examples/` (OO API) and `ext/examples/` (procedural API) are runnable, tested examples, not just documentation — keep them working when changing either API surface.
